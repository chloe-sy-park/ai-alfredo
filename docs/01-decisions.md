# 01. 모순 해결 결정사항

> 설계 과정에서 발견된 모순들과 해결 방안

---

## 🔴 모순 1: 온보딩 질문 수

### 문제
- **기존 설계**: "1개 질문만" (ADHD 친화적)
- **충돌**: 데이터 프라이버시 레벨 선택도 필요

### 결정
**데이터 레벨은 기본값 "균형"으로 자동 설정**

- 온보딩에서는 도움 유형 1개 질문만
- 데이터 레벨은 "균형"으로 자동 시작
- 프라이버시 민감 사용자는 설정에서 변경

### 이유
ADHD 사용자에게 온보딩은 짧을수록 좋음. 대부분의 사용자는 기본값을 유지하고, 민감한 사용자만 나중에 조정.

## 🔴 모순 2: 뷰 시스템 중복

### 문제
- **기존**: 홈/워크/라이프 3개 페이지 존재
- **신규**: 뷰 전환 시스템 (워크/라이프/통합)
- **충돌**: 기능 중복

### 결정
**홈에서 뷰 전환, 페이지는 심화 역할**

```
홈 = 대시보드 (워크/라이프/통합 뷰 전환 필터)
워크 페이지 = 업무 심화 (프로젝트 상세, 미팅 관리)
라이프 페이지 = 일상 심화 (습관 상세, 웰니스 트래킹)
```

### 이유
홈에서 빠른 전환, 깊은 관리는 전용 페이지에서. 중복 없이 역할 분리.

## 🔴 모순 3: 육성 4축 vs 프라이버시 레벨

### 문제
- **육성 시스템**: 4축 (톤, 알림, 동기부여, 데이터 깊이)
- **프라이버시 시스템**: 3단계 (최소/균형/풀)
- **충돌**: "데이터 깊이" 축과 프라이버시 레벨 중복

### 결정
**프라이버시 레벨로 통합, 육성은 3축으로 단순화**

- 육성 시스템에서 "데이터 깊이" 축 제거
- 육성은 **3축**으로 단순화: 톤 / 알림 / 동기부여
- 프라이버시는 별도 설정으로 분리 (설정 > 개인정보)

### 이유
같은 개념 두 곳에서 관리하면 혼란. 프라이버시는 민감한 설정이므로 별도 관리가 적절.

## 📋 결정 요약

| 모순 | 결정 | 핵심 원칙 |
|------|------|----------|
| 온보딩 질문 수 | 기본값 자동 설정 | 짧은 온보딩 |
| 뷰 시스템 중복 | 홈=필터, 페이지=심화 | 역할 분리 |
| 육성/프라이버시 | 프라이버시 별도 분리 | 단순화 |
